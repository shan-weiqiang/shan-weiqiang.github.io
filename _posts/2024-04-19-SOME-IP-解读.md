[SOME/IP](https://www.autosar.org/fileadmin/standards/R22-11/FO/AUTOSAR_PRS_SOMEIPProtocol.pdf) 协议是AUTOSAR AP和CP标准中包含的面向服务的通信协议。在AP上，SOME/IP 基于网络传输层的TCP和UDP协议，属于OSI模型的应用层协议。本文不是原文的翻译，而是按照章节对重要的，且难以理解的部分进行解释，希望能帮助到有需要的读者。

> SOME/IP在AUTOSAR中是通信的基础，在Manifest模型中，非常多的模型定义都跟SOME/IP相关；理解SOME/IP对理解AUTOSAR模型系统很重要

# 4 Protocol Specification

## 4.1 Specification of SOME/IP Message Format (Serialization)

### 4.1.2 Header

#### 4.1.2.1 Message ID [32 Bit]

- 消息ID全局唯一，唯一的确定一个Event或者Method
- 由两部分组成，分别占[16 bits]：
    - Service ID：所属的service ID
    - Method ID：这里Method ID可以是服务内部的Methods或者Events的ID。名称有误导性

#### 4.1.2.4 Request ID [32 Bit]

这个字段跟Method，field中的setter，getter(特殊的Methods)相关，因为同一个Skeleton提供的Methods可能同时并发多个请求方，这些请求方的传入参数是不同的，需要区分这些不同的请求。

> 在具体实现中，Method的处理一般是采用线程池并发处理请求
> 

这个ID由两部分组成，分别占[16 bits]：

- Client ID [16 Bits]
- Session ID [16 Bits]

其中，Client ID是全局唯一的。Session ID可以看作是当前Client对当前方法的请求计数器。一个Request ID完全由Proxy端产生，并且需要保证全局唯一性；Skeleton端无需关心这个ID，只需要在回复时将此ID拷贝到消息中即可，这包含了下面的结论：

- Proxy端在使用一个Request ID前必须保证该ID在上一次请求中已经返回结果或者超时，或者从未被使用，否则Skeleton端返回的请求结果可能跟Proxy端的请求对不上

> 关于Client ID： AUTOSAR文档中只介绍全局唯一，但并没有详细解释其来源。其实这是很重要的一个概念，与SOA通信的架构有关。在面向服务的通信模式中，一个服务提供方(Skeleton)有一个全局唯一的服务ID，称为Instance ID；服务ID是由Skeleton确定的。一个Skeleton可以同时提供服务给多个消费方(Proxy)，这些Proxy都需要指定其需要的具体的一个服务ID(或者所有服务ID）；当不同的Proxy所需要的服务ID是同一个时，那么对于这个服务实例来说，它就有多个Client，那么如何区分这不同的Client，AUTOSAR并没有一个Proxy服务实例的ID规定。在AUTOSAR模型系统中，唯一能确定一个Proxy的是一个全局唯一的模型元素路径，这个全局唯一的路径可以映射成全局唯一的Client ID。
> 

> Client ID除了用在Request/Response模式中，还可以用在Event的Publish/Subscribe方式中；例如同一个Host上有多个订阅同一个外部服务ID的Client，可以只在Host上建立一条与服务提供方的通信链路，由中心节点来通过不同的Client ID是否订阅来确定是否将Event消息转发给对应的Client，因为不同Client可能订阅的Event是不同的。有了Client ID，订阅同一个服务ID的不同Clients才可能使用同一个通信链路(UDP或者TCP端口号)，否则Proxy Host这一侧无法判断是否要将一个Event消息转发给不同的Client。
> 

#### 4.1.2.7 Message Type [8 Bit]

消息类型在SOME/IP协议中预留了位置来指定，在DDS的绑定中需要在实现时添加相关消息头信息。

- 所以DDS协议作为底层驱动时更加的灵活，没有SOME/IP这种协议级别的负担

### 4.1.4 Serialization of Data Structures

#### 4.1.4.3 Structured Datatypes and Arguments with Identifier and optional members (’TLV’)

这里有个Data ID和tag的概念：

- 用于数据类型结构体的成员前
- 用于Method入参前

功能是为了前后的兼容性，有了Data ID，接收方可以根据已知的Data ID去选择性的处理相关的数据。Data ID的位置决定了它必须在一个结构体内部或者一个方法的所有参数范围内唯一。有了Data ID后SOME/IP还支持将数据类型信息(预先定义的枚举值）与Data ID一起序列化到消息中，称为消息的tag

> SOME/IP相比其他通信协议，如DDS，Google Protobuf等有一个优势是支持union类型；这在车载通信与MCU交互时用的很多
> 

## 4.2 Specification of SOME/IP Protocol

### 4.2.1 Transport Protocol Bindings

- SOME/IP支持TCP或者UDP传输层协议
- 如果一个Machine上同时提供多个同一个Service ID的不同Instance，则这些Instance必须被绑定到不同的Port口，无论是TCP还是UDP，详细原因分析看4.2.1.3
- 一个Service Instance可以使用的传输层通信方式：
    - TCP单播
    - UDP单播
    - UDP组播
    
    > 这三个Port口是可以同时存在的，但是每一个最多只能有一个；对于服务下面的Event或者Method，只能同时对应一个TCP单播或者UDP单播，不能-》同时使用两个
    > 
    

#### 4.2.1.1 UDP Binding

- Skeleton和Proxy服务实例之间只能唯一的使用一个UDP的单播通信连接进行所有的Event，Method通信

#### 4.2.1.2 TCP Binding

- Skeleton和Proxy服务实例之间只能唯一的使用一个UDP的单播通信连接进行所有的Event，Method通信
- 在使用TCP传输层协议时，必须关闭Nagle算法，保证每次数据发送都必须马上打包成一个TCP segment发送出去，目的是保证消息消息实时性。
  
    > Nagle算法将多个TCP发送积累成一个大的TCP segment发送的算法，目的是提高TCP数据包的payload比例；因为一个TCP报文的头部占40-Byte，如果单个segment的payload过小，则发送的数据大部分都是头部信息，降低了报文的利用率，增加了线上阻塞的风险
    > 
- TCP的链接发起、重连、关闭都是由Proxy端决定的
    - 当Skeleton端调用了StopOfferService接口停止服务后，不能自行终断TCP链接，而应该是Proxy端来操作，否则Proxy端会进行重连，造成资源浪费

#### 4.2.1.3 Multiple Service-Instances

这里原文解释的已经很清楚，但由于逻辑关系的复杂性和重要性，这里有必要做一个更加详尽的解释。

- 首先一个Host，可以针对同一个Service提供多个不同的Service Instance，即服务的实例
- 每一个Service Instance在一个Service ID下有一个全局唯一的Instance ID
- 已知SOME/IP的消息头（不是SOME/IP-SD的消息头，这是服务发现的消息）是不包含Service Instance ID信息的
- 以上就会导致一个问题：客户端(Proxy, 或者Client）如何通过SOME/IP消息头来判断这个消息来自同一个Server上的同一个Service的哪一个Instance

有了问题的原因，让我们一步一步分析和解决这个问题。无论是UDP还是TCP都有一个(srcIP, srcPort, dstIP, dstPort)的tuple来表示一个通信链接：

- 不同Host上的服务实例，无论是同一个服务还是不同的服务，都可以通过通信链接来区分不同的Instance
- 同一个Host上的不同的Service，可以通过其唯一的Service ID来区分
- 同一个Host上的相同的Service，因为它们有相同的Service ID，而且SOME/IP消息头不包含Instance ID的信息，只能通过通信链接来区分不同的Instance；从一个Client Host的视角来看srcIP, dstIP是不能变动的，那么能区分的就只有srcPort或者dstPort，而Instance ID是由服务的提供方，即Skeleton侧定义的，所以区分也自然应该由Server端来区分：
    - 同一个Host上相同的Service的不同Instance需要绑定到不同的Port端口

反之：

- 如果一个Client Host上有同一个服务的不同服务实例的订阅，可以通过Skeleton端不同的端口号来区分这些服务实例
- 如果同一个Instance ID的多个Proxy，因为多个Proxy实例可以订阅同一个Instance ID提供的服务，即一发多收，则也可以通过同一个Port口来接收，因为每一个Client都有一个全局唯一的Client ID，当这个Instance ID通过唯一的通信链接到达Client Host时可以通过SOME/IP消息头部的Client ID信息来确定是由哪一个Proxy实例来接收

> 注意：这个要求只存在于Skeleton，即服务的提供方一侧
> 

#### 4.2.1.4 Transporting large SOME/IP messages of UDP (SOME/IP-TP)

针对一个UDP报文放不下的数据，SOME/IP使用SOME/IP-TP（SOME/IP Transport Protocol）来分成多个UDP报文发送，基本的原理就是每一个分割后的UDP报文都自带一个Session ID接收端通过Session ID来进行组装。SOME/IP标准的建议是能使用UDP的地方尽量使用UDP，只有在传输很大的数据，且没有严格的延迟要求的情况下才使用TCP

### 4.2.2 Request/Response Communication

- 这里有一个要求，所有Request的参数必须按照在函数调用中的实际参数位置顺序进行序列化

> Request和Response在落地时，使用C++的`std::future/std::promise`来异步实现
> 

### 4.2.3 Fire&Forget Communication

- 与普通Request/Response的区别在与不会等待Response

> 这里有个很大的误区就是认为Fire&Forget就是返回值为`void` 的特殊情况，其实这是错误的。当然Fire&Forget返回值类型是`void`，但是与普通Request/Response返回值是`void`不同，Fire/Forget不会等待，所以无法确定，其请求的任务是否执行完毕。但是普通的返回`void`的情况下，虽然返回值是`void`，但是可以通过`std::future`的`get()`方法来判断服务方是否已经执行完毕，这种情况往往需要服务方发送完成的标志位给到请求方。而Fire&Forget情况下，服务提供方是不会发送任何回复给请求方的，这也是为什么Fire&Forget的请求永远不会返回错误码。
> 

### 4.2.4 Notification Events

AUTOSAR要求当一个Proxy Host上有多个订阅同一个服务ID的Client时，实现上需要考虑降低网络流量，这其实确定了SOME/IP的实现往往需要一个中心节点。只有通过这个中心节点，做统一的收发管理，才能最大程度的优化网络流量。

#### 4.2.4.1 Strategy for sending notifications

订阅发送的三种模式：

- 周期发送
- 事件触发发送
- 值变化超过某个范围发送

> 实际项目中这些发送逻辑往往是用户来实现的
> 

### 4.2.5 Fields

Fields是Events和Methods的融合体，具体表现在：

- 一个Fields需要至少包含一个`getter`，一个`setter`，一个`notifier`
- `getter`和`setter`，本质就是Methods
- `notifier`，本质就是Event

# 6 Protocol usage and guidelines

## 6.1 Choosing the transport protocol

SOME/IP的传输层协议是具体到一个消息的：

- 每一个Event，Method等都可以配置自己的传输层协议

在UDP和TCP的选择上，建议如下：

- >1400Bytes，且在错误发生时没有严格实时性要求的，使用TCP
- 如果在错误发生时实时性要求<100ms，使用UDP
- 如果错误发生时实时性要求<100ms，且消息>1400 Bytes，使用SOME/IP-TP
